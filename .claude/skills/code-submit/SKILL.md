# Code Submit Skill

## 🎯 Skill 目标
该 Skill 用于在**人工审核已经完成后**，对当前代码变更进行提交前整理与收口，确保：

- 本次代码改动说明清晰
- 相关文档与说明文件已同步更新
- 测试 / 验证信息完整
- 提交信息规范统一
- 提交内容具备可追溯性与可交付性

该 Skill 适用于：
- 功能开发完成后的代码提交整理
- Bug 修复完成后的代码提交整理
- 已通过人工审核后的最终提交准备
- PR 前的最终收口

---

## 📚 强制文档同步更新（代码变更后必须执行）

### 适用范围
当代码发生任何变更时（包括但不限于新增功能、Bug 修复、重构、配置调整），必须同步检查并更新相关文档与说明文件。已归档文档除外。

---

### 必须更新的内容

#### 1. `docs/` 目录
需要根据本次代码改动检查并更新：

- 功能说明文档（新增 / 修改功能）
- API / 数据结构说明
- 使用方式 / 示例代码
- 设计说明（如影响架构、流程或逻辑）

如果本次变更会影响使用者、维护者或后续开发者理解系统行为，则必须同步更新 `docs/`。

#### 2. `AGENTS.md`
当以下情况发生时，必须同步更新 `AGENTS.md`：

- 新增能力或 Skill
- 修改现有流程、约束或行为边界
- 新增工具依赖或调用方式
- 调整 Agent 的职责、输出要求或协作方式

必须更新的内容包括：
- 能力描述
- 调用条件
- 输出规范
- 行为约束
- 与其他 Skill / Agent 的协作关系

---

### 更新原则
所有文档更新必须遵循以下原则：

- 与当前代码行为保持一致
- 覆盖主路径与关键边界情况
- 避免只更新局部说明导致整体理解偏差
- 明确体现新增、修改或废弃内容
- 保持文档结构与术语一致性

---

### 输出要求
在完成提交整理时，必须明确输出：

1. 已更新的文档文件列表
2. 每个文档的更新摘要
3. 若某类文档未更新，需说明“不适用”的原因
4. 若代码有行为变化但未更新文档，必须先补充文档再继续提交整理

---

## 📥 输入要求
在执行该 Skill 时，应尽可能从当前上下文、仓库状态与本次改动中整理出以下信息：

- 本次任务类型（feature / fix / refactor / docs / test / chore）
- 需求 / Bug / Issue / Jira（如有）
- 修改范围（模块 / 文件 / 功能点）
- 是否存在 Breaking Change
- 是否新增或修改测试
- 是否新增或修改文档
- 是否涉及 `AGENTS.md`
- 测试与验证结果
- 需要纳入提交说明的关键背景

如果部分信息上下文中未明确，也应基于当前代码与文件变更进行总结。

---

## 📦 提交整理范围

该 Skill 的职责包括但不限于：

### 1. 整理代码变更
输出本次提交涉及的：
- 核心功能改动
- 修复点
- 重构点
- 配置或结构调整
- 影响范围

---

### 2. 整理文档变更
输出本次提交涉及的：
- `docs/` 更新内容
- `AGENTS.md` 更新内容

---

### 3. 整理测试与验证结果
输出本次改动对应的验证情况，包括：

- 单元测试
- 集成测试
- E2E / Playwright
- Smoke Test
- 手动验证（如适用）

若测试结果不完整，应明确指出缺失项。

---

### 4. 生成规范的提交信息
根据本次变更内容生成：

- 推荐 commit type
- 推荐 commit message
- 推荐提交摘要
- 推荐 PR 标题（如需要）
- 推荐 PR 描述（如需要）

---

### 5. 提供风险与注意事项
明确说明：
- 潜在影响范围
- 是否存在兼容性风险
- 是否建议拆分提交
- 是否需要额外回归验证

---

## 🧩 Commit Message 规范（建议）
默认建议使用 Conventional Commits：

- `feat:` 新功能
- `fix:` Bug 修复
- `refactor:` 重构
- `docs:` 文档更新
- `test:` 测试补充
- `chore:` 杂项维护

推荐格式：

`type(scope): summary`

例如：

- `feat(renderer): add region-aware loading logic`
- `fix(config): handle default region fallback`
- `docs(skill): update AGENTS and docs for commit workflow`

---

## 📝 提交说明生成要求

在完成整理后，必须生成以下内容：

### 1. 变更摘要
说明本次改动的目标、原因与主要内容。

---

### 2. 文件分类清单
至少按以下类别输出：

- 代码文件
- 文档文件
- 测试文件
- Agent / Skill 相关文件

---

### 3. 文档更新摘要
说明：
- 更新了哪些 `docs/`
- 是否更新了 `AGENTS.md`
- 每项更新对应解决了什么问题

---

### 4. 测试结果摘要
说明：
- 执行了哪些测试
- 测试是否通过
- 覆盖了哪些主路径 / 边界场景
- 是否仍有未覆盖风险

---

### 5. 推荐提交信息
输出：
- 推荐 commit message
- 可选 PR 标题
- 可选 PR 描述

---

### 6. 风险与回滚提示
说明：
- 哪些模块受影响
- 如果出现问题应如何快速定位或回滚
- 是否需要重点关注上线后的观察项

---

## ✅ 执行原则

### 1. 文档更新是提交内容的一部分
代码变更、文档变更、验证材料变更应视为同一交付的一部分，不可分离。

---

### 2. 优先保证提交内容完整
在生成提交信息前，应先保证：
- 代码改动总结完整
- 文档同步完整
- 测试结果完整
- 风险说明完整

---

### 3. 保证输出可直接复用
Skill 的输出应尽量可直接用于：
- commit message
- PR 描述
- 交付说明
- Jira / Issue 更新说明

---

### 4. 优先输出结构化结果
建议输出顺序为：

1. 结论摘要
2. 代码变更摘要
3. 文档更新摘要
4. 测试结果摘要
5. 推荐 commit message
6. 推荐 PR 描述
7. 风险与注意事项

---

## 🔗 与其他 Skill 的关系

### 与 `dev` skill 配合
- `dev` 负责需求实现、测试、CR 与集成验证
- `code-submit` 负责最终提交整理、文档同步与提交说明生成

### 与 `bug-fix` skill 配合
- `bug-fix` 负责复现、定位、修复与验证
- `code-submit` 负责修复完成后的提交材料整理

### 与 `research` 配合
当本次变更引用了参考项目分析结论时，应在提交说明中体现：
- 使用了哪些参考分析
- 新增了哪些调研文档
- 是否同步更新了 `AGENTS.md`
