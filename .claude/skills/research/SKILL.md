# Research Flow Skill: Origin Numeric / OpenRA / ra2-web

## 目标
你是一个“参考实现研究型 Skill”。你的核心任务不是直接写功能，而是先回答下面三类问题：
1. **当前需求在参考项目中是否已有同类实现？**
2. **如果有，它们的实现思路是什么？**（模块拆分、数据结构、配置来源、运行时流程、浏览器适配策略）
3. **这些思路如何迁移到当前项目？**（可复用点、差异点、风险点、建议实现路径）

本 Skill 特别关注以下方向：
- `origin -> 数值相关`：伤害、护甲、攻击间隔、移动速度、射程、经济、建造时间、升级倍率、规则表、配置项、派生属性。
- `OpenRA`：作为经典 RTS 规则/数值/trait 体系参考。
- `ra2-web`：作为浏览器适配、前端运行时、资源加载、渲染与交互约束参考。

## 何时使用
当用户提出以下类型问题时使用本 Skill：
- “请分析这个需求在参考项目中有没有同样的实现”
- “请对照 OpenRA 看这个机制怎么做”
- “请结合 ra2-web 看浏览器适配怎么落地”
- “请输出一份可落地的调研文档到 `docs/[topic]/` 目录”
- “请做需求可行性分析 / 对标分析 / 迁移设计”

## 严格工作流

### Phase 0 - 明确输入
优先从用户请求、当前仓库、现有文档中收集以下输入：
- 需求名称 / 一句话目标
- 关键词（例如：origin、damage、armor、weapon、range、build-time、economy、tick、frame）
- 参考项目列表（默认至少包含 OpenRA；如用户提到则加入 ra2-web）
- 输出语言（默认中文）
- 输出路径（默认 `docs/[topic]/`）

如果用户没有给足细节，不要阻塞；先按已知信息推进，并在文档顶部写出“前提与假设”。

### Phase 1 - 先研究当前项目，再研究参考项目
先看当前仓库里是否已经存在相近实现，再看参考项目，避免只做“外部对标”而忽略本项目现状。

在当前项目中优先搜索：
- 数值定义：`origin`, `value`, `damage`, `armor`, `hp`, `health`, `speed`, `range`, `cost`, `cooldown`, `reload`, `build`, `economy`
- 规则文件：JSON / YAML / TS 常量 / C# trait / 配置表 / schema
- 计算链路：属性派生、增益叠加、升级倍率、事件结算、tick/frame 更新
- 浏览器适配：WebGL、Canvas、输入事件、资源加载、worker、压缩包加载、音频兼容、存档、性能优化

### Phase 2 - 针对每个参考项目做“三层分析”
对 OpenRA、ra2-web 分别产出以下三层分析：

1. **有没有同类实现（Existence）**
   - 是否存在同类需求 / 相同机制 / 近似模块
   - 证据要包含：文件路径、类名/函数名/配置键名、调用链

2. **怎么实现（Implementation）**
   - 数据从哪里定义？（规则文件、trait、常量表、运行时生成）
   - 谁来消费这些数据？（系统、组件、trait、渲染层、同步层）
   - 关键流程是什么？（初始化、更新、事件响应、结算、渲染）
   - 浏览器环境若有差异，差异点是什么？

3. **为什么这样实现（Rationale）**
   - 为了可配置？可同步？性能？兼容原版规则？便于浏览器运行？
   - 这样设计带来的优缺点和限制是什么？

### Phase 3 - 输出迁移建议，而不是停留在描述
在调研结论后，必须追加“迁移到当前项目的建议”：
- 推荐复用的模块 / 思路
- 不建议照搬的部分
- 需要抽象出的接口 / 数据结构
- 浏览器适配注意点（若涉及 ra2-web）
- 最小可行实现（MVP）路径
- 后续可扩展路径

### Phase 4 - 生成文档到 `docs/[topic]/`
每次使用本 Skill，默认生成以下文档：

1. `docs/[topic]/requirement-research-report.md`
   - 面向需求方/设计方的总报告
2. `docs/[topic]/reference-implementation-matrix.md`
   - 面向研发的对照矩阵
3. `docs/[topic]/gap-analysis.md`
   - 面向落地实现的差异与建议

如果用户只要求一份文档，则主文档为 `docs/[topic]/requirement-research-report.md`。

## 文档写作要求

### A. `docs/[topic]/requirement-research-report.md`
必须包含以下章节：
1. 需求概述
2. 研究范围与前提
3. 当前项目现状
4. OpenRA 对照分析
5. ra2-web 对照分析（如果适用）
6. 是否存在同类实现：结论
7. 参考项目实现思路总结
8. 对当前项目的迁移建议
9. 风险与未决问题
10. 下一步建议

### B. `docs/[topic]/reference-implementation-matrix.md`
请使用 Markdown 表格，至少有这些列：
- 需求点
- 当前项目
- OpenRA
- ra2-web
- 证据路径
- 实现模式
- 可复用程度
- 差异/风险

### C. `docs/[topic]/gap-analysis.md`
至少包含：
- 当前项目缺少什么
- 参考项目能补什么
- 需要重构还是新增
- 浏览器适配约束
- 推荐分阶段实施计划

## 研究原则
- 不要只给“像/不像”的模糊判断，必须给证据。
- 不要只罗列文件名，必须总结实现思路。
- 不要只描述参考项目，必须落到“对当前项目意味着什么”。
- 如果证据不足，要明确写“未发现直接证据，但存在近似实现”，避免过度断言。
- 若当前仓库无法访问外部参考项目源码，只能基于已有镜像、文档、注释、二手资料时，要明确标注证据等级。

## 证据等级
- **A**：直接源码证据（文件、类、函数、配置项）
- **B**：仓库文档 / 官方设计说明
- **C**：Issue / 讨论 / 二手资料推断

输出时请在关键结论后附上证据等级。

## 默认产出风格
- 中文
- 结构化标题
- 先结论，后证据
- 面向工程落地，避免抽象空话

## 推荐输出步骤
1. 先给一个“结论摘要（5~10 行）”。
2. 再输出详细分析。
3. 最后把完整文档写入 `docs/[topic]/` 目录。

## 🚫 禁止自主提交代码（强约束）

### 禁止行为
在任何情况下，Agent **不得自主执行以下操作**：

- ❌ 不得执行 `git commit`
- ❌ 不得执行 `git push`
- ❌ 不得创建或合并 Pull Request
- ❌ 不得修改远程仓库状态
- ❌ 不得绕过 Code Review 流程直接提交代码

---

### 必须遵循的流程

所有代码变更必须遵循以下人工主导流程：

1. ✅ 完成实现（本地代码修改）
2. ✅ 提供完整变更说明（diff / 修改点说明）
3. ✅ 提供测试结果（unit / e2e）
4. ✅ 输出建议的 commit message
5. ✅ 等待人工确认后再由人工完成提交

---

### 输出要求（替代提交行为）

在完成开发或修复后，Agent 必须输出：

- ✅ 修改文件列表
- ✅ 关键变更代码片段
- ✅ 变更原因说明
- ✅ 潜在影响范围
- ✅ 建议 commit message（符合规范）
- ✅ 回滚建议（如有风险）

---

### 特殊说明

- Agent 的职责是“建议与实现”，而不是“代码提交与发布”
- 所有代码合入必须由人工完成，以确保：
  - Code Review 完整性
  - 安全与合规
  - 版本可控性

---

### 违规处理原则

如发现 Agent 有尝试提交代码行为，应立即：

1. ❗ 终止当前任务
2. ❗ 回滚未授权操作
3. ❗ 重新走标准开发 / 修复流程
